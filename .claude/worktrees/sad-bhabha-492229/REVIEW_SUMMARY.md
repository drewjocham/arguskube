## 1. Architecture Review

### 1.1 Strengths

- Multi-layered pattern generally sound. Clean separation of Vue 3 frontend from Go backend via Wails bindings (`runtime.EventsEmit` / Go method calls)
- Good use of `SharedInformerFactory` in the in-cluster agent (`agent/internal/k8s/client.go`)
- Background event loop with configurable polling intervals (5s alerts, 10s metrics, 3s logs) at `api/pkg/events.go`
- Strong concurrency patterns: context-based cancellation, errgroup usage throughout
- MCP (Model Context Protocol) server with 12 tools is well-structured
- Context assembler (`internal/context/assembler.go`) with parallel gathering, timeouts, and graceful partial results is genuinely well-designed

### 1.2 Critical Issues

| Issue | Details | Location |
|---|---|---|
| **Desktop-SaaS conflation** | `api/pkg/` mixes desktop Wails bindings with SaaS HTTP server logic. These should be separate packages or modules. | `api/pkg/app.go` + `api/pkg/server.go` |
| **Security: Agent auth** | No mTLS for WebSocket tunnel connections. Agent registration/auth model is unclear. No certificate validation strategy. | `api/pkg/hub.go`, `agent/internal/tunnel/client.go` |
| **Security: Agent RBAC** | In-cluster agent ServiceAccount permissions undefined. Could be over-permissioned. | `agent/` |
| **Security: SaaS API auth** | No authentication middleware on SaaS HTTP endpoints. | `api/pkg/server.go` |
| **Scalability: Hub bottleneck** | Agent connection management uses `map[string]*Agent + sync.Mutex` — will become a bottleneck with many agents. | `api/pkg/hub.go:21-38` |
| **Scalability: Multi-tenant SaaS** | No tenant isolation model, no message broker (NATS/RabbitMQ), no API gateway. Current design doesn't scale horizontally. | SaaS architecture not yet designed |
| **No tests** | Zero test files found in the entire codebase. No CI/CD pipeline configured. | — |
| **Monolithic resource file** | `internal/k8s/resources.go` at 1817 lines handles 17 resource types in one file. Violates SRP. | `internal/k8s/resources.go` |
| **Dead dependencies** | `go-chi/chi/v5` imported but unused (uses stdlib routing). BadgerDB imported but not actively used. | `go.mod` |
| **Event loop + Wails lifecycle** | Background polling continues even when Wails window is closed. No pause-on-hide logic. | `api/pkg/events.go` |

### 1.3 Architecture-Design Mismatch

The architecture doc describes a **desktop-light, agent-heavy** design where the agent scrapes metrics, handles logs, and runs ML inference. Reality is the exact opposite — the **desktop client does everything** via `client-go` REST calls:

- Metrics: Desktop calls metrics-server proxy directly (`internal/k8s/metrics.go:66-122`) instead of agent scraping Kubelet `/metrics/cadvisor`
- Logs: Desktop does `client-go` GetLogs REST streaming (`internal/k8s/logs.go:117-158`) instead of agent hooking Kubelet log files
- Anomaly Detection: Anomstack client is client-side, not agent-deployed
- Agent connector (`internal/agentconn/connector.go`) exists but the agent itself is near-empty scaffolding with no WebSocket streaming endpoints

**Recommendation**: Either make the agent actually do what the architecture doc says, or update the architecture doc to reflect the desktop-heavy reality.

### 1.4 SaaS Infrastructure Recommendations

For a multi-tenant SaaS deployment on GCP:

1. **Microservices decomposition**: Split `api/pkg/` into: API Gateway, Tenant Service, Agent Hub Service, Metrics Service
2. **PubSub for agent comms**: Replace direct WebSocket hub with NATS/RabbitMQ for reliable message delivery at scale
3. **Tenant isolation**: Database-per-tenant or row-level security in PostgreSQL
4. **Agent auth**: JWT-based registration with rotating tokens + mTLS for tunnel connections
5. **API Gateway**: Use Envoy or Kong for rate limiting, auth, routing
6. **Observability stack**: Prometheus + Grafana + distributed tracing (Jaeger/Tempo)
7. **CI/CD**: GitHub Actions for build/test, ArgoCD for GitOps deployment

### 2.4 Module Strategy

Two `go.mod` files (`backend/` + `agent/`) is correct separation. Risk: dependency drift over time.

**Recommendation**: Use a shared `internal/` types module or establish a CI check that ensures compatible versions via `go vet` and regular `go mod tidy` runs on both modules.

### 3.5 One-Click Philosophy Violations

From AGENTS.md: *"If a feature requires more than two clicks to achieve its primary goal, it is a failure."*

1. **Pod deletion**: 4 clicks (expand → Delete → confirm → confirm). Add right-click context menu for direct delete.
2. **Vulnerability scan**: 3+ clicks. Should auto-scan on view load with background status indicator.
3. **Apply VPA recommendation**: Stub only, but even if real it would be 3+ clicks for what should be one-click.
4. **Context switching**: 3+ clicks (sidebar → cluster → context → wait). Should be one-click from titlebar.

### 3.6 AI Integration Quality

**Genuinely useful**: The diagnosis engine (`internal/context/diagnose.go`) produces actionable outputs — ready-to-paste `kubectl` commands, concrete resource recommendations, pattern-matched diagnosis for 6 alert types (OOM, CrashLoop, DiskPressure, CPUThrottle, ImagePull, MemPressure).

**Misleading**: It's a **rules engine** with hardcoded confidence scores, not actual AI. DeepSeek (`internal/ai/deepseek.go`) is **not wired** to the context assembler or diagnosis flow. The "AI-powered" label on the diagnostics panel is disingenuous.

**Fix**: Wire DeepSeek to receive the context bundle and generate a genuine LLM-powered summary alongside the rule-based diagnosis. This validates the "AI" branding with minimal effort.
