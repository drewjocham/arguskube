  9.1 Architecture & Patterns

    ✓ OAuth PKCE flow following RFC 8252 (native apps)
    ✓ Centralized error types with HTTP disposition mapping

  IMPROVEMENT AREAS:

    [G1] Reflective HTTP dispatch is fragile
      server.go uses reflect.ValueOf(a).MethodByName() — any public method
      signature change silently breaks API compatibility.
      Suggestion: explicit handler registration via interface or code-gen.

    [G2] Multiple K8s client implementations
      Two independent K8s client layers: internal/k8s (desktop) and pkg/kube (MCP).
      Shared types exist (pkg/kube/types.go) but no shared client abstraction.
      Suggestion: unify under a shared client interface.

    [G3] Polling instead of watching
      events.go uses 5s/10s/3s polling for alerts/metrics/logs.
      Suggestion: use K8s informers/watch API for real-time updates.

    [G4] Spot-check engine is untested
      spotcheck/spotcheck.go and checks.go have zero test files.
      Suggestion: at minimum, unit-test the individual Check implementations.

    [G5] App.go is a god struct
      App struct embeds 20+ dependencies. NewApp constructor is very long.
      Suggestion: split into domain-specific sub-structs or use interface segregation.

    [G6] Missing password.go tests
      bcrypt hash/verify has no dedicated test file.
      Suggestion: add table-driven tests for hash + verify + cost + edge cases.

    [G7] agentanalysis is a no-op
      agent.go has a 3-second sleep as "analysis" — placeholder for future work.
      Suggestion: either implement real analysis or remove the stub.

    [G8] MCP CORS is wide open
      mcp/api/api.go uses AllowAll() CORS — contrast with backend's restrictive policy.
      Suggestion: align with KUBEWATCHER_API_ALLOWED_ORIGINS when possible.
