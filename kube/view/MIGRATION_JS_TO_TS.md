# JS → TS Migration Plan (`kube/view`)

## State at plan time

- **165 `.js` files** in `kube/view/src` (excluding `wailsjs/` generated bindings and `node_modules/`).
- **25 existing `.ts` files** — terminal, profiles, output capture, bus, tasks, a few composables. The infrastructure is already TS-aware.
- `tsconfig.json` has `allowJs: true, checkJs: false` — JS files pass through unchecked. The migration is purely additive (no breakage as files convert).
- The `wailsjs/` directory is **excluded** from this migration. It's generated by Wails at build time and ships `.d.ts` files already.

## Principles

1. **One phase per PR.** Each PR can be merged independently; the codebase always builds at any commit.
2. **Migrate leaves first.** Pure functions before things that import them, so `tsc` errors stay localized.
3. **No semantic changes.** Type annotations only. If a `.js` file has a bug, fix it in a separate PR — don't bundle it with the conversion.
4. **`unknown` over `any`.** Inputs that genuinely can be anything are typed `unknown` and narrowed at the use site. `any` is allowed only for third-party shapes we can't reasonably model in one pass.
5. **Wailsjs types are gospel.** When a composable calls into `window.go.pkg.App.X`, the result type comes straight from `src/wailsjs/go/pkg/App.d.ts` — no remodeling.
6. **Tests convert with their source.** A `.test.js` is renamed alongside its source `.js` so a `git log -- path` still reads cleanly.
7. **Don't enable `strict` mode globally yet.** Each phase can opt the converted files in by adding them to a per-file override or pinning specific flags. Strict-mode rollout is a final, separate phase.

## Phases

### Phase 1 — `src/utils/` (this PR)
Pure functions, zero Vue/Pinia dependencies, zero backend coupling. 5 source + 5 test files. Smallest possible blast radius — proves the migration mechanics work before touching anything reactive.

### Phase 2 — `src/lib/`
4 source + 1 test. Bus, tasks, plus a couple of small helpers. Mostly already typed informally via JSDoc.

### Phase 3 — `src/stores/`
26 source + 13 test. Pinia stores. Most have implicit shape via initial state — the conversion documents what each ref/computed actually is. Some stores call into composables, so this lands AFTER composables only IF a store ref needs a composable type; otherwise do it in parallel.

### Phase 4 — `src/composables/`
31 source + 25 test. Bridge calls to `window.go` (Wails) and `fetch` (SaaS HTTP shim) — most of the typing work is here because the wire types come from `wailsjs/go/models.ts`. Plan for ~3 sub-PRs:
  - 4a: pure-state composables (no callGo) — useToast, useBackgroundTasks, etc.
  - 4b: read-only callGo composables — useResources, useAlerts, …
  - 4c: mutating/streaming composables — useLogs, useExec, …

### Phase 5 — `src/__tests__/components/`
41 component test suites. Each one mounts a Vue SFC + asserts. Conversion is mostly mechanical (rename + add explicit types for `mount` returns).

### Phase 6 — Stragglers
`src/features/`, `src/__tests__/{stores,composables,lib}`, the one `.js` under `src/storage/`, anything else surfaced by `find`.

### Phase 7 — Tighten
- Flip `checkJs: false → true` if any `.js` survives (none should at this point).
- Adopt `"strict": true` on the converted paths via TS project references or `include` overrides.
- Add a CI check that fails on new `.js` introductions.

## Per-file conversion recipe

1. `git mv foo.js foo.ts`
2. Run the existing tests (`npx vitest run path/to/foo.test.ts`) — they should pass unchanged with `allowJs: true`.
3. Add type annotations only where TS infers `any`. Lean on inference; don't over-annotate.
4. Replace JSDoc `@typedef` blocks with real `interface` / `type` declarations.
5. If a function takes/returns a backend-derived shape, import the type from `wailsjs/go/models`.
6. Run `npx vue-tsc --noEmit` against the file's directory to verify it type-checks.
7. Run the full Vitest suite — no behavior change should regress.
